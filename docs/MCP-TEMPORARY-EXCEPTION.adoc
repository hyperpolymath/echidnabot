= Temporary Native MCP — BoJ-Only Exception
:toc: left
:toclevels: 2

== Status

**TEMPORARY.** The estate-wide rule is _"all MCP tools route through the
BoJ server"_ (see memory `feedback_boj_only_mcp`). This binary
(`echidnabot-mcp`) is a standalone MCP server, so it is a documented
deviation from that rule, not a revision of it.

== Why this exists

The BoJ V-lang adapter at
`boj-server/adapter/v/boj-server` was removed during the V-lang ban
(2026-04-10). The Zig replacement at `boj-server/adapter/zig/` is an
empty source directory — not started. The BoJ server is therefore
unreachable on this workstation: its SessionStart health-check hook
(`~/.claude/hooks/boj-health-check.sh`) fails three times every session
with `BoJ V-lang adapter not compiled` and graceful-degrades.

Package 7 of the echidnabot actuation plan needs MCP exposure so
echidnabot is directly callable by AI assistants (Claude Desktop /
Claude Code, etc.) to invoke `suggest_tactics`, record outcomes, and
trigger corpus refresh. Waiting on the BoJ Zig rewrite would
indefinitely block Package 7 items 5 and the rest of the double-loop
wiring. So: native MCP ships now under an explicit exception.

== Exception scope

* Applies to this single binary: `echidnabot-mcp`.
* Does **not** green-light new standalone MCP servers elsewhere in the
  estate. Every other service remains BoJ-cartridge-only.
* Expires when the BoJ Zig adapter lands.

== Retirement plan

When the BoJ Zig adapter is rebuilt:

. Port this binary's tool surface (the `suggest_tactics`,
  `record_tactic_outcome`, `list_outcome_history`, `corpus_refresh`
  handlers in `src/bin/mcp_server.rs`) into a BoJ cartridge at
  `boj-server/cartridges/echidna-llm-mcp/` (which already exists as a
  thin wrapper and can absorb the richer surface).
. Remove the `[[bin]] echidnabot-mcp` entry from `Cargo.toml`.
. Delete `src/bin/mcp_server.rs` and this file.
. Update `~/.config/claude/mcp_servers.json` to re-route via BoJ (it
  already does — BoJ is configured, just non-functional).
. Delete the project memory
  `project_boj_mcp_temporary_exception.md`.

The tool logic itself (reranker + store + corpus-delta writer) stays in
the library crate and is re-consumed by the cartridge.

== Usage

=== Build

[source,console]
----
cd verification-ecosystem/echidnabot
cargo build --bin echidnabot-mcp --release
----

Binary ends up at `target/release/echidnabot-mcp`.

=== Configuration

Reads `echidnabot.toml` (overridable via `ECHIDNABOT_CONFIG` env) plus
all `ECHIDNABOT__*` environment variables, same as the main binary.

Minimum viable config (writable SQLite URL):

[source,toml]
----
[database]
url = "sqlite:///var/lib/echidnabot/state.db?mode=rwc"

[echidna]
endpoint = "https://echidna.hyperpolymath.dev/graphql"
rest_endpoint = "https://echidna.hyperpolymath.dev"
mode = "auto"
timeout_secs = 300

# Optional: enable corpus-delta writes + retrain trigger
[corpus]
enabled = true
training_data_dir = "/var/mnt/eclipse/repos/verification-ecosystem/echidna/training_data"
echidna_root = "/var/mnt/eclipse/repos/verification-ecosystem/echidna"
auto_trigger_threshold = 50   # optional
----

Database note: sqlx will not create a SQLite file without `?mode=rwc`
in the URL. Run `echidnabot init-db` first, or bake the mode parameter
into the config as shown above.

=== Claude Code / Claude Desktop

Add to the client's MCP-servers config (NOT `~/.config/claude/mcp_servers.json`
— that one is reserved for BoJ). For Claude Desktop:

[source,json]
----
{
  "mcpServers": {
    "echidnabot": {
      "command": "/path/to/target/release/echidnabot-mcp",
      "env": {
        "ECHIDNABOT_CONFIG": "/etc/echidnabot.toml"
      }
    }
  }
}
----

=== Protocol

Transport: JSON-RPC 2.0, newline-delimited, over stdio. Stderr is used
for logs; stdout is the protocol channel only.

Tools exposed:

[cols="1,3"]
|===
|Tool |Purpose

|`suggest_tactics`
|Request tactics from ECHIDNA, reranked against local history.

|`record_tactic_outcome`
|Persist an attempt outcome. Feeds the reranker + (if enabled)
emits a row to the corpus delta.

|`list_outcome_history`
|Query the `tactic_outcomes` store by goal-fingerprint or tactic.

|`corpus_refresh`
|Fire `just corpus-refresh` in `corpus.echidna_root`. Errors if
corpus is disabled.
|===

== Schema-gap caveat (7b-3)

`corpus_refresh` today fires the retrain pipeline, but the delta rows
this bot emits do not yet match `echidna/scripts/merge_corpus.jl`'s
hardcoded `PER_PROVER_FILES` whitelist and schema (see STATE.a2ml
`double-loop-feedback > known-gap`). Until that bridge lands,
`corpus_refresh` runs the pipeline against whatever existing corpora
the echidna repo has, and ignores this bot's delta files. Coordination
ticket with Session A.
